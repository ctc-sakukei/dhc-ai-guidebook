# 基本知识
了解以下知识可以更好的理解/使用Claude Code。
## Prompt (提示词)
给AI模型的输入指令或问题。好的Prompt应该清晰、具体，包含足够的上下文信息。例如"用Python写一个排序函数"比"写代码"更有效。

## Token
AI模型处理文本的基本单位，类似于"字块"。1个Token约等于0.75个英文单词或0.5个中文字符。Token数量影响成本和处理速度。
## Temperature (温度)
控制AI输出随机性的参数，范围0-1。值越低输出越确定和保守，值越高输出越有创意和多样性。代码生成通常使用较低温度。
## Hallucination (幻觉)
AI模型生成看似合理但实际错误或虚构的内容。常见于编造不存在的API、函数名或事实。使用时需要验证关键信息的准确性。
## Context Window (上下文窗口)
AI模型一次能处理的最大Token数量。Claude Sonnet 4.5的窗口为200K Token，能处理约15万字的中文内容，包括对话历史和输入文档。

# 进阶内容
通用的AI知识，了解这些概念有助于理解AI应用的工作原理，在定制开发或遇到问题时会更容易理解背后的机制。

## System Prompt / User Prompt
System Prompt设定AI的角色和行为规则（如"你是专业的代码助手"），User Prompt是用户的具体问题或指令。两者配合引导AI输出。

## Agent
能够自主规划和执行多步骤任务的AI系统。Agent可以调用工具、读写文件、运行代码，直到完成目标。Claude Code就是一个典型的Agent。

## Copilot
辅助型AI助手，根据当前上下文提供实时建议和补全。主要用于代码编写，如GitHub Copilot在编码时自动补全代码片段。

## Agent VS Copilot
Agent主动完成完整任务（如"实现登录功能"），Copilot被动提供建议（如自动补全代码）。Agent更自主，Copilot更轻量快速。

## Tool Calling / Function Calling
AI模型调用外部工具或函数的能力。例如Claude Code可以调用文件读写、命令执行等工具，从而完成代码修改、运行测试等实际操作。

## Streaming (API是否以Stream的形式返回)
流式返回指AI逐字输出内容，用户能实时看到生成过程，体验更流畅。非流式则等待全部内容生成完毕后一次性返回。

## Latency (延迟)
从发送请求到收到AI响应的时间。影响因素包括模型大小、网络速度、Token数量。Streaming模式可以降低感知延迟，提升用户体验。

## MCP
Model Context Protocol，AI模型与外部数据源连接的标准协议。通过MCP，AI可以访问数据库、API、文件系统等资源，扩展能力边界。

## RAG
Retrieval-Augmented Generation，检索增强生成。先从知识库检索相关信息，再结合这些信息生成回答，可有效减少幻觉，提供更准确的答案。

### Embedding
将文本转换为数字向量的技术，语义相似的文本向量距离更近。RAG系统用Embedding来快速找到与问题最相关的知识片段。

# Claude Code
## Claude
Anthropic公司开发的大语言模型系列，以安全性和可靠性著称。支持长上下文理解、多模态输入，擅长复杂推理和代码生成任务。
## Claude Code
基于Claude的AI编程助手，是一个完整的Agent系统。可以自主读写文件、执行命令、搜索代码，帮助开发者完成从需求分析到代码实现的全流程开发任务。
## Claude 提供的模型
Claude提供三个层级的模型：Opus（最强大）、Sonnet（平衡性能与成本）、Haiku（快速轻量）。根据任务复杂度和预算需求选择合适的模型。
### Opus 4.6
最强大的Claude模型，擅长处理复杂推理、架构设计等高难度任务。适用于需要深度思考和精确输出的场景，成本最高但能力最强。
### Sonnet 4.6
性能和成本平衡的Claude模型，是Claude Code的默认模型。能够高效处理日常开发任务，响应速度快，适合大多数编程场景。
### Haiku 4.5
最快速、成本最低的Claude模型。适合简单重复性任务、快速代码补全等轻量场景。延迟低，响应迅速，但处理复杂任务能力相对较弱。
## 其他的AI编程助手
市场上还有GitHub Copilot、Cursor、Windsurf等AI编程工具。Copilot侧重代码补全，Cursor和Windsurf提供类似Claude Code的Agent能力，各有特色。

## 如何选择合适的Claude模型

在实际使用中，需要根据项目规模和具体需求选择合适的模型：

### 大参数模型（如Opus、Sonnet）

**优点：**
- 对庞大的项目支持更好，能够理解复杂的代码库结构
- 可以处理复杂的业务逻辑和架构设计
- 支持更多的开发语言和技术栈
- 推理能力更强，能够处理更抽象的问题

**缺点：**
- 响应速度较慢，生成内容需要更多时间
- 使用成本较高（按Token计费）
- 本地部署难度大或无法本地部署（需要大量计算资源）

### 小参数模型（如Haiku及其他轻量级模型）

**优点：**
- 响应速度快，适合需要快速反馈的场景
- 使用成本低，适合大量简单任务
- 可以本地部署，对显存要求相对较低
- 适合标准化、模式化的任务

**缺点：**
- 处理复杂业务逻辑时容易出错或遗漏细节
- 主要支持主流的开发语言和技术，对小众技术栈支持有限
- 对大型项目的整体把握能力较弱
- 处理抽象概念和复杂推理时表现不足

### 选择建议

- **企业级大型项目**：推荐使用Opus或Sonnet，确保代码质量和理解深度
- **日常开发任务**：Sonnet提供了最佳的性能与成本平衡
- **简单重复任务**：可以使用Haiku或其他轻量级模型，如代码格式化、简单重构等
- **成本敏感场景**：优先使用Haiku处理简单任务，仅在必要时切换到更强大的模型
- **本地部署需求**：考虑使用开源的小参数模型，但需要接受其性能限制
