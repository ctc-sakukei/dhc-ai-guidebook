# Claude Code 进阶指南

本文介绍成本控制、安全注意事项和常见问题解决方案。

## 成本控制策略

由于Claude服务按Token计费，合理控制成本对长期使用至关重要。

### Token消耗来源

1. **上下文（输入）**：每次对话包含历史记录
2. **生成（输出）**：CC生成的所有内容
3. **文件读取**：每次读取文件消耗Token
4. **重复工作**：因错误导致的返工

### 成本优化策略

#### 策略1：及时清理上下文

```bash
# 完成任务后立即清空
/clear

# 定期压缩
/compact
```

**节省效果**：可减少50%以上的上下文Token消耗

#### 策略2：精确指定文件范围

```
❌ 成本高：请阅读整个src目录

✅ 成本低：请阅读src/services/UserService.java
```

**节省效果**：避免不必要的文件读取，节省30-50% Token

#### 策略3：使用缓存策略

对于大型任务，使用文件缓存而不是依赖上下文：

```
你：请测试case1-3，结果保存到./temp/result1.md
你：/clear
你：请测试case4-6，结果保存到./temp/result2.md
你：/clear
你：请读取所有result文件生成总结报告
```

**节省效果**：对于10个以上测试用例的场景，可节省70% Token

#### 策略4：选择合适的模型

| 任务类型 | 推荐模型 | 成本 |
|---------|---------|------|
| 简单代码生成 | Haiku | 低 |
| 日常开发 | Sonnet | 中 |
| 复杂架构设计 | Opus | 高 |

**节省效果**：合理使用Haiku可节省60-80%成本

#### 策略5：一次性说清楚

```
❌ 多次对话（成本高）：
你：生成一个用户类
CC：[生成基础类]
你：加上验证
CC：[添加验证]
你：再加上toString
CC：[添加toString]

✅ 一次说清（成本低）：
你：生成用户类，包含id、name、email字段，
    添加javax.validation验证，重写toString
CC：[一次生成完整代码]
```

**节省效果**：减少70%的对话轮次

### 成本监控

**定期检查Token使用：**
- 查看Claude账户使用统计
- 记录大型任务Token消耗
- 分析哪些操作最耗费Token

**设置预算提醒：**
- 为项目设定月度Token预算
- 接近预算时更谨慎使用CC

## 安全注意事项

在使用CC时，安全问题不容忽视。

### 不要跳过授权确认

CC设置中有`Allow Dangerously Skip Permissions`选项。关闭时，CC执行命令前会要求授权。

#### 为什么不能跳过

许多命令可能产生危险后果：

- `npm install`、`./gradlew bootRun`等构建命令
- 文件删除命令（`rm`、`del`）
- 数据库操作命令

**特别警告**：WSL中的危险命令（如`rm -rf /`）也会影响Windows文件系统，可能导致严重数据丢失。

#### 正确做法

CC请求执行命令时：

1. **仔细阅读**命令的具体内容
2. **理解**命令会做什么操作
3. **确认**命令的影响范围
4. 只有在完全理解并确认安全后，才点击授权

> **重要**：永远不要盲目授权，也不要为了方便启用"跳过权限确认"。安全永远比便利更重要。

## 常见问题FAQ

### Q1: CC生成的代码有bug怎么办？

**A:**
1. 不要直接删除，先让CC自己review找问题
2. 明确指出bug的具体表现
3. 如果CC多次修复失败，考虑人工修复或换思路
4. 将常见bug记录到Memory，避免重复

```
你：这段代码在处理null值时会抛出NullPointerException，
    具体在第15行的obj.toString()。请修复。
```

### Q2: 如何让CC遵守项目代码规范？

**A:**
1. 在CLAUDE.md中明确记录代码规范
2. 在Prompt中重申规范
3. 提供示例代码作为参考
4. 使用linter工具自动检查

```
你：请按照项目规范生成代码：
- 使用4空格缩进
- 变量使用驼峰命名
- 所有public方法必须有JavaDoc
- 参考现有的UserService.java风格
```

### Q3: 上下文满了但任务还没完成怎么办？

**A:**
1. 使用`/compact`压缩上下文
2. 使用缓存策略（见[3-ClaudeCode核心概念.md](3-ClaudeCode核心概念.md#上下文管理策略)）
3. 将任务拆分为更小的子任务
4. 使用Sub-Agent处理大型任务

```
你：请将测试结果保存到./temp/test-results.md
你：/clear
你：请读取./temp/test-results.md生成总结报告
```

### Q4: CC一直在做错误的事情怎么纠正？

**A:**
1. 立即停止：使用`/clear`清空上下文
2. 重新开始，使用更明确的指令
3. 提供正确的示例
4. 分解任务，逐步验证

```
你：/clear
你：我重新说明需求。我需要的是...（详细说明）
    请先告诉我你的理解，不要直接动手
```

### Q5: 如何让CC理解复杂业务逻辑？

**A:**
1. 分层次说明：先整体，再细节
2. 使用流程图或伪代码
3. 提供实际案例
4. 参考现有代码

```
你：订单流程如下：
1. 用户提交订单
2. 系统检查库存
   - 如果库存充足：锁定库存，进入步骤3
   - 如果库存不足：提示用户，流程结束
3. 调用支付接口
   - 如果支付成功：扣减库存，创建订单，进入步骤4
   - 如果支付失败：释放库存，提示用户，流程结束
4. 发送订单确认邮件

请基于这个流程实现OrderService.createOrder方法
```

### Q6: CC生成的测试用例不够全面怎么办？

**A:**
1. 明确列出需要覆盖的场景
2. 让CC分析可能的边界条件
3. 参考现有的测试风格

```
你：请为login方法生成测试，需要覆盖：
1. 正常登录（用户名密码正确）
2. 用户名不存在
3. 密码错误
4. 用户被锁定
5. 参数为null
6. 参数为空字符串
参考UserServiceTest.java的测试风格
```

### Q7: 如何提高CC的响应速度？

**A:**
1. 减少上下文长度（及时清理）
2. 明确指定文件范围
3. 避免让CC读取大文件
4. 使用`/fast`模式（牺牲质量换速度）
5. 使用Haiku模型处理简单任务

### Q8: CC修改了不该改的文件怎么办？

**A:**
1. 使用git回退：`git checkout -- file.java`
2. 在IDE中使用撤销功能
3. 下次明确说明只修改哪些文件

```
你：请只修改UserService.java文件，不要动其他文件
```

### Q9: 如何让团队成员更好地使用CC？

**A:**
1. 创建详细的CLAUDE.md
2. 共享Memory配置
3. 建立团队使用规范
4. 定期分享最佳实践
5. 记录常见问题和解决方案

### Q10: CC能完全替代人工开发吗？

**A:**

**不能。** CC是强大的辅助工具，但无法替代人类开发者：

- ✅ CC擅长：生成样板代码、编写测试、重构、文档生成
- ❌ CC不擅长：复杂架构设计、业务决策、创新性解决方案
- ⚠️ 必须人工：代码review、安全审查、性能优化、生产部署

**正确定位**：将CC视为高效的助手，而不是完全的替代品。

## 总结

掌握以上成本控制、安全注意事项和常见问题解决方案，你就可以高效、安全地使用Claude Code了。

**关键要点回顾：**

1. **成本控制**：及时清理上下文、精确指定范围、选择合适模型
2. **安全第一**：永远不要跳过权限确认
3. **分步实施**：复杂任务拆分为小步骤
4. **及时Review**：不要盲目信任AI输出
5. **持续优化**：记录经验到Memory，不断改进工作流程
