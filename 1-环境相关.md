# 工作流程
## 同一系统（推荐）

### 使用IDE插件

这是**最便捷的使用方式**，适合日常开发。

**VSCode示例：**
1. 在扩展市场安装 `Claude Code` 插件
2. 打开项目文件夹
3. 点击侧边栏Claude Code图标或按 `Ctrl+Shift+P` 唤起
4. 与AI对话，文件修改会实时显示

**优点：**
- 操作体验流畅，无需切换应用
- AI能直接看到你在IDE中选中的代码
- 文件修改实时同步，所见即所得

### 使用命令行

适用于无法安装插件的情况（如受限的企业环境）。

**VSCode示例：**
1. 打开项目
2. 按 `` Ctrl+` `` 打开集成终端
3. 运行 `claude-code`
4. 在终端与AI对话，文件修改后IDE自动刷新

**优点：**
- 不依赖IDE插件，适用范围广
- AI仍然能够修改文件，只是需要手动刷新IDE查看

**注意事项：**
- 需要在终端和IDE之间切换
- 部分IDE可能需要手动刷新才能看到文件变化

## 跨系统使用

典型场景：Windows上使用IDE，Claude Code运行在WSL中。

### 方法一：使用IDE远程开发功能（推荐）

通过IDE的远程连接功能，直接在远程系统中开发，体验如同本地。

**VSCode示例：**
- **连接到WSL**：安装 `Remote - WSL` 扩展 → 点击左下角绿色按钮连接 → 打开项目 → 运行 `claude-code`
- **连接到远程服务器**：安装 `Remote - SSH` 扩展 → 配置SSH主机 → 连接并打开项目 → 运行 `claude-code`
- **连接到Docker容器**：安装 `Dev Containers` 扩展 → 连接到运行中的容器 → 运行 `claude-code`

**优点：**
- **最佳体验**：IDE完全运行在远程环境，无需文件同步
- **性能优秀**：避免跨文件系统访问，速度快
- **环境统一**：编辑器、终端、Claude Code都在同一系统中
- **支持插件**：可以在远程环境中安装和使用Claude Code插件
- **适用场景广**：支持WSL、SSH远程服务器、Docker容器等

**注意事项：**
- 首次连接需要在远程系统安装VSCode Server（自动完成）
- 确保远程系统已安装Claude Code CLI
- 部分扩展需要在远程环境中重新安装
- WSL模式下推荐将项目放在WSL文件系统（`~/`）而非Windows挂载目录（`/mnt/c/`）

### 方法二：直接文件系统访问

适合实时查看AI修改的场景。

**VSCode示例：**
- Windows中打开项目：`C:\projects\my-project`
- WSL终端中运行：`cd /mnt/c/projects/my-project && claude-code`
- VSCode自动检测文件变化

**优点：**
- 可以利用WSL的Linux环境优势（更好的命令行工具、包管理器等）
- 适合需要Linux环境编译/运行的项目

**注意事项：**
- 确保IDE和WSL访问的是同一个项目目录
- Windows路径 `C:\path\to\project` 对应 WSL路径 `/mnt/c/path/to/project`
- 注意文件系统权限问题
- WSL和Windows文件系统间的性能差异（建议项目放在WSL文件系统中以获得更好性能）

### 方法三：通过Git同步

适合完全隔离的环境，或希望保持清晰的代码版本管理。

**VSCode示例：**
- **WSL中开发**：运行 `claude-code` → AI完成后提交 → `git push`
- **Windows中同步**：在VSCode源代码管理面板点击"拉取" 或运行 `git pull`
- **查看修改**：通过提交历史查看AI的变更

**优点：**
- 完全隔离的开发环境，避免文件系统兼容性问题
- 清晰的版本控制，每次AI修改都有提交记录
- 适合WSL项目在独立文件系统中（如 `~/project`）的情况
- 可以利用Git分支进行实验性开发

**注意事项：**
- 需要频繁执行 git pull/push 操作
- 适合以"任务"为单位的开发模式（完成一个功能后再同步）
- 确保两边都配置了Git，且有远程仓库访问权限
- 注意 `.gitignore` 配置，避免同步不必要的文件

## 如何选择？

| 场景 | 推荐方案 | 适用人群 |
|------|---------|---------|
| 日常开发 | IDE插件方式 | 所有用户 |
| 企业受限环境 | 同系统命令行方式 | 无法安装插件的用户 |
| 跨系统开发（推荐） | IDE远程开发 | 需要Linux环境或远程服务器 |
| 简单跨系统 | 跨系统直接访问 | 临时使用，快速上手 |
| 需要版本管理 | 跨系统Git同步 | 希望清晰的提交历史 |

**新手建议：**
- 同系统开发：优先使用IDE插件
- 跨系统开发：优先使用IDE远程开发功能（如VSCode Remote-WSL）
