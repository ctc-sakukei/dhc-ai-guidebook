# Claude Code 使用技巧

本文介绍与Claude Code对话的实用技巧和最佳实践，帮助你更高效地使用CC。

## 对话技巧精华

### 技巧1：明确开发者的预设

#### 问题

开发者思考时基于隐含预设（项目知识、术语缩写），但CC不知道这些预设，容易产生意外结果。

#### 典型案例

假设项目使用"XXX"代表特定数据对象。直接说"请生成一个XXX"，CC可能：

1. 回复"不知道XXX是什么"
2. **危险**：自行理解并执行SQL创建数据，污染数据库

#### 解决方案

```
❌ 错误：请生成一个XXX

✅ 正确：请生成一个XXX（完整术语：用户配置项），
        包含字段：userId、configKey、configValue
```

**最佳实践：**
- 在`CLAUDE.md`中定义项目术语
- Prompt中明确说明具体含义
- 使用Memory保存常用术语

### 技巧2：避免模棱两可的Prompt

#### 问题

开发者基于预设思考，AI考虑更多可能性，容易偏离意图。

#### 典型案例

```
你说：让Header的颜色是红色

你的想法：#FF0000正红色，修改.header的background-color

AI的疑问：
- Header是什么？页面标题？导航栏？某个组件？
- 红色是什么？正红？深红？暗红？番茄红？

实际结果：可能给id为header的div加了style="#8B0000"
```

#### 解决方案

**精确到值的级别：**

```
❌ 模糊：让Header的颜色是红色

✅ 清晰：将class为.header的background-color设置为#FF0000

❌ 模糊：增加一个用户名字段

✅ 清晰：在user_table表中增加varchar(100)类型的name字段，
        not null，默认值为空字符串
```

### 技巧3：先总结，再执行

#### 问题

直接说"这个问题请解决"，CC会自动排查并修改，存在：

1. **缺乏透明度**：不知道CC如何修改
2. **方案可能偏离**：CC的方案可能不是你期望的
3. **撤销困难**：事后修正容易出问题

#### 解决方案

**三步法：**

```
步骤1 - 分析：
你：数据库更新失败的原因是什么？请分析可能的原因

步骤2 - 提案：
你：请提供2-3个解决方案，说明每个方案的优缺点

步骤3 - 执行：
你：我选择方案2，请实施
```

**好处：**
- 了解改修细节
- 方便代码review
- 节省时间和Token（避免返工）

### 技巧4：保证设计书的正确性

#### 问题

设计书错误会导致恶性循环：

1. 基于错误设计生成代码
2. 测试发现问题
3. 让CC根据设计书调查 → 大概率查不出（设计书本身错了）
4. 开发者亲自调查 → 发现设计书错误
5. 修改设计书，重新生成代码

**后果**：严重浪费时间和Token。

#### 解决方案

**设计书质量保证：**

- **人工Review**：编写后先人工审查
- **AI辅助Review**：让CC检查逻辑矛盾
- **详细说明**：关键业务逻辑详细说明
- **术语一致性**：确保术语和规格一致

### 技巧5：警惕AI幻觉

#### 什么是AI幻觉

AI生成的答案实际上错误，但AI以确信的语气表达，让人误以为正确。

#### 应对方法

- **保持怀疑**：不要完全信任AI答案
- **人工Review**：关键代码必须审查
- **测试验证**：充分测试生成的代码
- **警惕完美答案**：太完美反而要警惕

#### 典型场景

容易出现幻觉：

- **API文档**：编造不存在的API方法
- **框架用法**：给出已废弃的旧版本用法
- **代码逻辑**：看起来正确但实际有bug

### 技巧6：控制单次输出长度

#### 问题

AWS Bedrock限制：**单次输出最大4096 Token**。

内容过长会被截断，导致不完整。

#### 容易触发的场景

- 大量代码生成（多个文件）
- 详细报告（测试报告、分析报告）
- 批量操作（多个相似任务）
- 完整文档（长设计文档）

#### 解决方案

**1. 任务拆分**

```
❌ 不推荐：请生成UserController、Service、Dao和所有测试

✅ 推荐：
步骤1：请先生成UserController
步骤2：现在生成UserService
步骤3：接下来生成UserDao
步骤4：最后生成测试代码
```

**2. 逐步处理**

```
❌ 不推荐：一次性测试所有10个用例并生成完整报告

✅ 推荐：
先测试用例1-3，保存到./temp/result1.md
再测试用例4-6，保存到./temp/result2.md
最后读取所有结果，生成汇总报告
```

**3. 明确输出范围**

```
❌ 不推荐：请详细分析这个模块的所有问题

✅ 推荐：
请先列出主要问题清单（不需要详细说明）
然后我会选择需要详细分析的问题
```

**4. 使用缓存策略**

结合[3-ClaudeCode核心概念.md](3-ClaudeCode核心概念.md#上下文管理策略)的缓存方法：

- 大任务分解为小步骤
- 每步输出保存到临时文件
- 完成后汇总所有文件

## 完整开发工作流示例

### 场景：添加用户注册功能

#### 第1步：需求确认

```
你：需要添加用户注册功能：
- 用户名（3-20字符，字母数字下划线）
- 邮箱（验证格式）
- 密码（至少8位，包含大小写字母和数字）
- 密码加密存储
- 邮箱唯一性验证
- 注册成功发送欢迎邮件

请先分析现有代码，给出实现方案

CC：[分析User实体、UserService等，给出方案]

你：方案2比较合适，请按方案2实施
```

#### 第2步：生成DTO

```
你：请先生成UserRegistrationDTO，包含用户名、邮箱、密码，
    并添加javax.validation注解

CC：[生成UserRegistrationDTO.java]

你：密码正则有问题，请改为：至少8位，包含大小写字母和数字

CC：[修改正则表达式]
```

#### 第3步：实现Service

```
你：在UserService中添加register方法：
1. 检查邮箱是否已存在
2. 使用BCrypt加密密码
3. 保存用户信息
4. 发送欢迎邮件（调用EmailService）
5. 返回用户ID

CC：[生成代码]

你：如果发送邮件失败，用户信息已保存怎么办？
    请提供2-3个解决方案

CC：[方案1:不回滚 2:事务回滚 3:异步发送]

你：采用方案3，异步发送邮件

CC：[修改为异步]
```

#### 第4步：实现Controller

```
你：在UserController中添加/api/users/register接口，
    调用register方法，返回标准响应格式

CC：[生成Controller]
```

#### 第5步：测试

```
你：为register方法生成单元测试，包括：
    正常注册、邮箱已存在、密码格式错误

CC：[生成测试]

你：请运行测试

CC：[运行测试，报告结果]
```

#### 第6步：调试

```
你：邮箱已存在的测试失败，NullPointerException。请调查

CC：[发现Mock配置有误，修复]

你：请再次运行测试

CC：[所有测试通过]
```

#### 第7步：Review和提交

```
你：review所有代码，检查：
1. 安全漏洞
2. 性能问题
3. 是否符合项目规范

CC：[Review并给出建议]

你：请按建议1和3修改

CC：[修改完成]

你：生成git commit，message要包含功能说明

CC：[创建commit]
```

### 工作流最佳实践

1. **明确需求**：详细说明功能和验收标准
2. **分步实施**：不要一次性生成所有代码
3. **及时Review**：每步完成后立即检查
4. **测试驱动**：生成代码后立即测试
5. **迭代改进**：发现问题立即修复
6. **代码审查**：提交前全面review

## 进阶技巧

### 技巧1：让CC分步思考

```
你：请分步分析这个性能问题：
1. 先找出可能的性能瓶颈
2. 对每个瓶颈分析原因
3. 提供优化方案
4. 评估每个方案的效果

CC：[按步骤逐一分析]
```

**优势**：思考更深入，答案更准确

### 技巧2：让CC自我检查

```
你：请生成快速排序算法

CC：[生成代码]

你：请review这段代码，检查：
1. 是否有bug
2. 边界条件是否正确
3. 是否有更优实现

CC：[自我审查并改进]
```

**优势**：提高代码质量，减少人工review

### 技巧3：链式Prompt

```
你：我要实现缓存系统，分三步：
第一步：分析需求并设计接口
第二步：实现核心功能
第三步：添加测试

现在开始第一步，请设计缓存接口

CC：[设计接口]

你：好的，进入第二步，实现核心功能

CC：[实现代码]
```

**优势**：保持连贯性，避免上下文混乱

### 技巧4：使用示例引导

```
你：请参考以下示例风格生成UserService：

示例（OrderService）：
@Service
public class OrderService {
    @Autowired
    private OrderRepository repository;

    public Order createOrder(OrderDTO dto) {
        // 参数验证
        // 业务逻辑
        // 保存数据
        // 返回结果
    }
}

现在用同样风格生成UserService

CC：[按示例风格生成]
```

**优势**：保持代码风格一致

### 技巧5：增量式重构

```
你：这个方法太长，请分步重构：
1. 先提取参数验证部分
2. 然后提取业务逻辑部分
3. 最后提取数据库操作部分
每步都要保证功能不变

CC：[逐步重构，每步确认]
```

**优势**：降低重构风险，便于验证
