# Claude Code 使用技巧

本文介绍与Claude Code对话的实用技巧和最佳实践，帮助你更高效地使用Claude Code。

在深入具体技巧之前，理解 Claude Code 的核心交互哲学至关重要  
高效的开发协同主要源于对以下三大原则的遵循
- **上下文余量优化**：合理规划上下文规模，确保模型始终保留充足的“推理空间”，避免因信息过载导致的响应质量下降。
- **任务原子化拆解**：将复杂、宽泛的工程需求解构为粒度更细、目标更具体的子任务，提升模型执行的准确性。
- **指令确定性准则**：向模型提供详尽、明确且具排他性的指令，通过减少语义模糊度来降低生成偏差。

## 配置与管理技巧

本章节介绍如何有效配置和管理 Claude Code 的核心功能，包括 CLAUDE.md、Skill/Agent 使用以及上下文管理。

### CLAUDE.md 遵守程度

Claude Code会**尽量遵守**CLAUDE.md规则，但**不是100%严格**。

**原因：**

当上下文接近容量上限需要压缩时，Claude Code可能判断部分CLAUDE.md内容优先级较低，将其从上下文中移除。

#### 应对策略

**策略1：关键规则重申**

对于特别重要的规则，在关键对话时再次强调：

```
你：请生成UserService，注意必须使用项目的BaseService
Claude Code：[遵守规则生成代码]
```

**策略2：Prompt中明确说明**

直接在指令中写明核心要求：

```
你：请按照以下规范生成代码：
- 使用4空格缩进
- 所有public方法必须有JavaDoc
- 参考UserService.java的风格
```

**策略3：保持CLAUDE.md简洁**

- 只保留最核心的规则
- 将详细说明放在其他文档
- 提高被保留在上下文的优先级

#### Agent 使用场景判断

**内置Agent类型：**

| Agent | 主要用途 | 特点 |
|-------|---------|------|
| `Explore` | 代码探索、文件搜索 | 快速定位代码 |
| `Plan` | 设计实现方案 | 架构规划 |

**何时使用Agent：**

✅ **推荐场景：**
- 需要大量文件搜索
- 上下文接近80%
- 并行处理独立任务
- 复杂代码生成/审查

❌ **不推荐场景：**
- 简单单文件读取
- 需要频繁交互确认

**使用示例：**


**最佳实践：**

- **保持简洁**：MEMORY.md只记录关键要点
- **使用链接**：详细内容放在专门文件中
- **定期更新**：及时记录新的决策和经验
- **按主题组织**：而非按时间顺序

## 上下文管理技巧

### 控制单次输出长度（重要）

#### 问题

由于CTC发现AWS Bedrock服务当单次输出超过4096会变得不稳定，所以强制增加了**单次输出最大4096 Token**的限制，并且**禁止开发人员擅自修改这个值**

带来的问题是，如果单次让Claude输出内容过多，会出现下面的错误信息
```
API Error: Claude's response exceeded the 4096 output token maximum. To configure this behavior, set the CLAUDE_CODE_MAX_OUTPUT_TOKENS environment variable.
```
出现这个信息之后，Claude Code会强制终止输出并且丢弃，用户不会获取到任何来自Claude的信息。相当于使用了Token，但是什么结果都没有得到

#### 容易触发的场景

主要是单次大量内容生成
- 直接生成整个详细设计
- 根据设计直接生成一个Service文件
- 将多个子报告合并成一个报告文件

#### 解决方案

**1. 任务拆分**

```
❌ 不推荐：请根据概要设计生成详细设计书

✅ 推荐：
步骤1：请根据概要设计书生成详细设计书的逻辑A部分
步骤2：请根据概要设计书生成详细设计书的逻辑B部分
...
```

**2. 逐步处理**

```
❌ 不推荐：请根据设计生成对应的Service代码

✅ 推荐：
步骤1：请根据设计书生成Service中逻辑A对应的方法
步骤1：请根据设计书生成Service中逻辑B 对应的方法
```

**3. 明确输出范围**

```
❌ 不推荐：请详细分析这个模块的所有问题

✅ 推荐：
请先列出主要问题清单（不需要详细说明）
然后我会选择需要详细分析的问题
```

**4. 使用缓存策略**

结合后续章节的缓存方法：
- 大任务分解为小步骤
- 每步输出保存到临时文件
- 完成后汇总所有文件

### 主动清理

```bash
# 完成任务后立即清空
/clear

# 上下文接近满时压缩
/compact
```

### 缓存策略

适用于复杂长任务，避免信息丢失：

**问题场景：**
测试10个用例，到第5个时上下文耗尽，最后生成报告时已"忘记"前5个结果。

**解决方案：**

```
步骤1：测试case1-3，结果保存到./temp/result1.md
步骤2：/clear
步骤3：测试case4-6，结果保存到./temp/result2.md
步骤4：/clear
步骤5：测试case7-10，结果保存到./temp/result3.md
步骤6：/clear
步骤7：读取所有result文件，生成汇总报告
```

**优势：**
- 避免上下文压缩导致的信息丢失
- 可处理任意规模的任务
- 中间结果持久保存

### Sub-Agent隔离

使用独立上下文的Agent处理大型子任务：

```
# 主对话上下文保持清洁
你：请用Explore agent全面搜索项目的认证实现

# Agent在独立上下文中工作
[Agent执行大量搜索，不占用主上下文]

# 返回精简结果到主对话
Claude Code：[汇总Agent发现的关键信息]
```

### 注意上下文的剩余空间

在Claude Code发现上下文空间被耗尽时可能会出现两种情况：

#### 情况1：触发强制压缩（大概率）

**表现：**
- 自动压缩历史对话内容，保留摘要
- 不会中断对话，可继续进行

**后果：**
- 丢失部分历史对话的详细内容
- 如果后续对话涉及到被压缩的内容，Claude Code生成的结果很可能与用户预期不符
- 摘要可能不够准确，导致理解偏差

#### 情况2：上下文继续提示

**表现：**
出现提示信息：
```
This session is being continued from a previous conversation that ran out of context.
The summary below covers the earlier portion of the conversation.
```

**含义：**
- 这是上下文已被压缩的提示
- 之前的详细对话已被总结成摘要
- 对话可以继续，但早期细节已丢失

**后果：**
- Claude Code会丢失大量对话内容的详细信息
- 早期对话的代码片段、具体讨论等无法准确回忆
- 如果摘要信息不完整，可能影响后续任务的准确性
- 可能需要重新提供关键信息

#### 应对策略

**预防措施：**
1. **定期检查**：注意Claude Code提示的上下文使用情况
2. **主动清理**：完成阶段性任务后使用`/clear`或`/compact`
3. **拆分任务**：将大任务分解为多个小任务
4. **使用缓存**：关键中间结果保存到文件中（参考[缓存策略](#缓存策略)）

**补救措施（出现提示后）：**
1. **评估影响**：判断丢失的信息是否影响当前任务
2. **重新提供**：将关键信息（需求、规范等）重新说明
3. **文件依赖**：依赖文件内容而非对话历史
4. **新建会话**：如果影响严重，考虑`/clear`后重新开始

## 对话技巧精华

### 明确开发者的预设

#### 问题

开发者思考时基于隐含预设（项目知识、术语缩写），但Claude Code不知道这些预设，容易产生意外结果。

#### 典型案例

假设项目使用"XXX"代表特定数据对象。直接说"请生成一个XXX"，Claude Code可能：

1. 回复"不知道XXX是什么"
2. **危险**：自行理解并执行SQL创建数据，污染数据库

#### 解决方案

```
❌ 错误：请生成一个XXX

✅ 正确：请生成一个XXX（完整术语：用户配置项），
        包含字段：userId、configKey、configValue
```

**最佳实践：**
- 在`CLAUDE.md`中定义项目术语
- Prompt中明确说明具体含义
- 使用Memory保存常用术语

### 避免二义性的Prompt

#### 问题

开发者基于预设思考，AI考虑更多可能性，容易偏离意图。

#### 典型案例

```
你说：让Header的颜色是红色

你的想法：#FF0000正红色，修改.header的background-color

AI的疑问：
- Header是什么？页面标题？导航栏？某个组件？
- 红色是什么？正红？深红？暗红？番茄红？

实际结果：可能给id为header的div加了style="#8B0000"
```

#### 解决方案

**精确到值的级别：**

```
❌ 模糊：让Header的颜色是红色

✅ 清晰：将class为.header的background-color设置为#FF0000

❌ 模糊：增加一个用户名字段

✅ 清晰：在user_table表中增加varchar(100)类型的name字段，
        not null，默认值为空字符串
```

### 先总结，再执行

#### 问题

直接说"这个问题请解决"，Claude Code会自动排查并修改，存在：

1. **缺乏透明度**：不知道Claude Code如何修改
2. **方案可能偏离**：Claude Code的方案可能不是你期望的
3. **撤销困难**：事后修正容易出问题

#### 解决方案

**三步法：**
1. 让AI先总结问题的原因
2. 让AI根据原因提供解决方案
3. 让AI执行
   * 如果此时上下文仍有剩余，可直接指定方案让AI执行
   * 如果上下文即将耗尽，指定一个方案让AI完整输出到硬盘上，开启新对话读取文件继续进行

比如
```
步骤1 - 分析：
你：数据库更新失败的原因是什么？请分析可能的原因

步骤2 - 提案：
你：请提供2-3个解决方案，说明每个方案的优缺点

步骤3 - 执行：
你：我选择方案2，请实施
```

**好处：**
- 了解改修细节
- 方便代码review
- 节省时间和Token（避免返工）

### 保证设计书的正确性

#### 问题

设计书错误会导致恶性循环：

1. 基于错误设计生成代码
2. 测试发现问题
3. 让Claude Code根据设计书调查 → 大概率查不出（设计书本身错了）
4. 开发者亲自调查 → 发现设计书错误
5. 修改设计书，重新生成代码

**后果**：严重浪费时间和Token。

#### 解决方案

**设计书质量保证：**

- **人工Review**：编写后先人工审查
- **AI辅助Review**：让Claude Code检查逻辑矛盾
- **详细说明**：关键业务逻辑详细说明
- **术语一致性**：确保术语和规格一致

### 警惕AI幻觉

#### 什么是AI幻觉

AI生成的答案实际上错误，但AI以确信的语气表达，让人误以为正确。

#### 应对方法

- **保持怀疑**：不要完全信任AI答案
- **人工Review**：关键代码必须审查
- **测试验证**：充分测试生成的代码
- **警惕完美答案**：太完美反而要警惕

#### 典型场景

容易出现幻觉：

- **API文档**：编造不存在的API方法
- **框架用法**：给出已废弃的旧版本用法
- **代码逻辑**：看起来正确但实际有bug

## 完整开发工作流示例

### 场景：添加用户注册功能

#### 第1步：需求确认

```
你：需要添加用户注册功能：
- 用户名（3-20字符，字母数字下划线）
- 邮箱（验证格式）
- 密码（至少8位，包含大小写字母和数字）
- 密码加密存储
- 邮箱唯一性验证
- 注册成功发送欢迎邮件

请先分析现有代码，给出实现方案

Claude Code：[分析User实体、UserService等，给出方案]

你：方案2比较合适，请按方案2实施
```

#### 第2步：生成DTO

```
你：请先生成UserRegistrationDTO，包含用户名、邮箱、密码，
    并添加javax.validation注解

Claude Code：[生成UserRegistrationDTO.java]

你：密码正则有问题，请改为：至少8位，包含大小写字母和数字

Claude Code：[修改正则表达式]
```

#### 第3步：实现Service

```
你：在UserService中添加register方法：
1. 检查邮箱是否已存在
2. 使用BCrypt加密密码
3. 保存用户信息
4. 发送欢迎邮件（调用EmailService）
5. 返回用户ID

Claude Code：[生成代码]

你：如果发送邮件失败，用户信息已保存怎么办？
    请提供2-3个解决方案

Claude Code：[方案1:不回滚 2:事务回滚 3:异步发送]

你：采用方案3，异步发送邮件

Claude Code：[修改为异步]
```

#### 第4步：实现Controller

```
你：在UserController中添加/api/users/register接口，
    调用register方法，返回标准响应格式

Claude Code：[生成Controller]
```

#### 第5步：测试

```
你：为register方法生成单元测试，包括：
    正常注册、邮箱已存在、密码格式错误

Claude Code：[生成测试]

你：请运行测试

Claude Code：[运行测试，报告结果]
```

#### 第6步：调试

```
你：邮箱已存在的测试失败，NullPointerException。请调查

Claude Code：[发现Mock配置有误，修复]

你：请再次运行测试

Claude Code：[所有测试通过]
```

#### 第7步：Review和提交

```
你：review所有代码，检查：
1. 安全漏洞
2. 性能问题
3. 是否符合项目规范

Claude Code：[Review并给出建议]

你：请按建议1和3修改

Claude Code：[修改完成]

你：生成git commit，message要包含功能说明

Claude Code：[创建commit]
```

### 工作流最佳实践

1. **明确需求**：详细说明功能和验收标准
2. **分步实施**：不要一次性生成所有代码
3. **及时Review**：每步完成后立即检查
4. **测试驱动**：生成代码后立即测试
5. **迭代改进**：发现问题立即修复
6. **代码审查**：提交前全面review

## 进阶技巧

### 技巧1：让Claude Code分步思考

```
你：请分步分析这个性能问题：
1. 先找出可能的性能瓶颈
2. 对每个瓶颈分析原因
3. 提供优化方案
4. 评估每个方案的效果

Claude Code：[按步骤逐一分析]
```

**优势**：思考更深入，答案更准确

### 技巧2：让Claude Code自我检查

```
你：请生成快速排序算法

Claude Code：[生成代码]

你：请review这段代码，检查：
1. 是否有bug
2. 边界条件是否正确
3. 是否有更优实现

Claude Code：[自我审查并改进]
```

**优势**：提高代码质量，减少人工review

### 技巧3：链式Prompt

```
你：我要实现缓存系统，分三步：
第一步：分析需求并设计接口
第二步：实现核心功能
第三步：添加测试

现在开始第一步，请设计缓存接口

Claude Code：[设计接口]

你：好的，进入第二步，实现核心功能

Claude Code：[实现代码]
```

**优势**：保持连贯性，避免上下文混乱

### 技巧4：使用示例引导

```
你：请参考以下示例风格生成UserService：

示例（OrderService）：

public class OrderService {
    @Autowired
    private OrderRepository repository;

    public Order createOrder(OrderDTO dto) {
        // 参数验证
        // 业务逻辑
        // 保存数据
        // 返回结果
    }
}

现在用同样风格生成UserService

Claude Code：[按示例风格生成]
```

**优势**：保持代码风格一致

### 技巧5：增量式重构

```
你：这个方法太长，请分步重构：
1. 先提取参数验证部分
2. 然后提取业务逻辑部分
3. 最后提取数据库操作部分
每步都要保证功能不变

Claude Code：[逐步重构，每步确认]
```

**优势**：降低重构风险，便于验证

## 常用场景Prompt模板

以下是10个高频开发场景的Prompt模板，可直接套用或根据实际情况调整：

### 1. 创建新功能

```
请实现[功能名称]功能：

需求说明：
- [具体需求1]
- [具体需求2]
- [具体需求3]

技术要求：
- 使用[技术栈/框架]
- 遵循[项目中的编码规范]
- 包含错误处理

交付物：
- [文件名1]：包含[具体内容]
- [文件名2]：包含[具体内容]
- 单元测试
```

### 2. 代码审查

```
请review以下代码文件：[文件路径]

重点检查：
1. 安全漏洞（SQL注入、XSS等）
2. 性能问题（N+1查询、内存泄漏等）
3. 代码规范（命名、注释、结构）
4. 边界条件处理
5. 错误处理是否完善

请分点列出问题，并对每个问题给出修改建议。
```

### 3. Bug调试

```
发现Bug：[简短描述问题]

现象：
- 预期行为：[描述]
- 实际行为：[描述]
- 复现步骤：[步骤]

相关代码：
- 涉及文件：[文件路径]
- 涉及方法/函数：[名称]

请按以下步骤进行：
1. 分析可能的原因（列出2-3个可能性）
2. 定位具体问题代码
3. 提供修复方案
4. 解释为什么会出现这个问题
```

### 4. 重构代码

```
需要重构：[文件路径或方法名]

重构原因：
- [问题1：如代码重复]
- [问题2：如逻辑复杂]
- [问题3：如可读性差]

重构要求：
- 保持功能完全一致
- 提高代码可读性
```
