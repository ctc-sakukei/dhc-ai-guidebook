# AI高级知识

本文档面向需要深入理解 AI 工作原理的开发者，特别是需要调用 AI API、开发 AI 应用或进行性能调优的技术负责人。

## Tool Calling / Function Calling

**概念说明**：
AI 模型调用外部工具或函数的能力。通过预定义的函数接口，AI 可以执行文件操作、命令执行、API 调用等实际操作。

**工作原理**：
```
1. 定义工具接口
{
  "name": "read_file",
  "description": "读取文件内容",
  "parameters": {
    "file_path": "string"
  }
}

2. AI 决策
用户：请修复 UserService 中的 Bug
AI 分析：需要先读取文件
AI 生成调用：read_file(file_path="/src/UserService.java")

3. 系统执行
执行工具并返回结果给 AI

4. AI 继续处理
基于文件内容进行下一步操作
```

**价值**：
- 突破纯文本限制，可以实际操作系统
- 自动选择合适的工具完成任务
- 多个工具可以组合使用解决复杂问题

**常见工具类型**：
- 文件操作：Read、Write、Edit
- 代码搜索：Grep、Glob
- 命令执行：Bash、Shell
- 网络请求：HTTP Client
- 数据库查询：SQL Executor

## Temperature (温度参数)

**概念说明**：
控制 AI 输出随机性的核心参数，取值范围 0-1。温度越低，输出越确定；温度越高，输出越随机。

**工作原理**：
```
AI 生成每个 Token 时计算概率分布：

原始概率（Temperature = 1.0）：
Token A: 40%
Token B: 30%
Token C: 20%
Token D: 10%

低温度（Temperature = 0.2）- 放大高概率选项：
Token A: 70%
Token B: 20%
Token C: 8%
Token D: 2%

高温度（Temperature = 1.5）- 平缓概率分布：
Token A: 35%
Token B: 28%
Token C: 22%
Token D: 15%
```

**应用场景**：

| Temperature | 输出特点 | 推荐场景 |
|------------|---------|---------|
| 0.0-0.3 | 高度确定、一致 | 代码生成、Bug 修复、数据提取 |
| 0.4-0.7 | 平衡确定性与多样性 | API 设计、架构讨论、技术文档 |
| 0.8-1.0 | 高创意、多样化 | 命名、头脑风暴、创意写作 |

**实战对比**：
```python
任务：写一个计算两数之和的函数

Temperature = 0.2（推荐代码生成）：
def add(a, b):
    return a + b
# 每次生成几乎相同

Temperature = 0.8（不推荐代码生成）：
# 第1次
def sum_numbers(x, y):
    result = x + y
    return result

# 第2次
def calculate_sum(num1, num2):
    """Adds two numbers"""
    return num1 + num2
# 每次生成不同，不利于代码一致性
```

## Top-P / Top-K（采样策略参数）

**概念说明**：
控制 AI 输出多样性的采样策略，与 Temperature 配合使用。Top-K 限制候选 Token 数量，Top-P 限制累积概率范围。

**Top-K 工作原理**：
```
Top-K = 3（只保留前3个最高概率的选项）

原始候选（10000+ 个 Token）：
Token A: 40%
Token B: 30%
Token C: 20%
Token D: 5%
...

应用 Top-K = 3 后：
✅ Token A: 40%
✅ Token B: 30%
✅ Token C: 20%
❌ Token D-Z: 全部排除

从 3 个候选中随机选择
```

**Top-P 工作原理**：
```
Top-P = 0.9（累积概率达到 90% 时停止）

Token A: 40% → 累积 40% ✅
Token B: 30% → 累积 70% ✅
Token C: 20% → 累积 90% ✅（达到阈值）
Token D: 5%  → 累积 95% ❌（超过阈值，排除）

动态候选池：根据概率分布自动调整
```

**参数组合最佳实践**：

| 任务类型 | Temperature | Top-P | 效果 |
|---------|------------|-------|------|
| 代码生成/修复 | 0.2-0.3 | 0.5-0.7 | 输出稳定准确 |
| 代码重构 | 0.3-0.5 | 0.7-0.9 | 保持一致性，适度创新 |
| API 设计 | 0.5-0.7 | 0.9 | 专业且有创意 |
| 命名/文档 | 0.7-0.8 | 0.9-1.0 | 多样化表达 |

## Streaming（流式输出）

**概念说明**：
AI 模型逐步返回生成内容的技术。与传统的等待完整响应不同，流式输出在生成第一个 Token 后立即开始返回。

**工作原理对比**：
```
非流式模式：
用户请求 → [等待3秒] → 完整响应一次性显示

流式模式：
用户请求 → [0.5秒] → 第一个 Token → 持续显示 → 完成
```

**价值**：
- 用户立即看到响应，降低感知延迟
- 可以提前阅读已生成内容
- 发现问题可及时中断，节省成本

**性能对比**：

| 模式 | 首次响应时间 | 用户感知延迟 | Token 成本 |
|------|------------|------------|----------|
| 非流式 | 慢（等待完整生成） | 高 | 相同 |
| 流式 | 极快（首 Token） | 低 | 相同 |

## Latency（延迟）

**概念说明**：
从发送请求到开始收到响应的时间间隔。延迟由网络延迟、队列等待、模型处理三部分组成。

**延迟组成**：
```
总延迟 = 网络延迟 + 队列等待 + 模型处理

网络延迟（10-200ms）
├─ 客户端到服务器
└─ 服务器到客户端

队列等待（0-5000ms）
├─ 高峰期排队
└─ 资源分配等待

模型处理（100-2000ms）
├─ 输入编码
├─ 首 Token 生成（TTFT）
└─ 后续 Token 生成
```

**关键指标**：

### TTFT（Time To First Token）
首 Token 时间，决定用户感知的响应速度

```
短上下文（<1K Tokens）：200-500ms
中等上下文（1K-10K Tokens）：500-1500ms
长上下文（10K-100K Tokens）：1500-3000ms
```

### TPS（Tokens Per Second）
每秒生成 Token 数，决定长文本生成速度

**延迟优化策略**：

| 优化方向 | 方法 | 效果 | 成本影响 |
|---------|------|------|---------|
| 减少上下文 | 只提供相关代码 | TTFT 降低 50% | 无 |
| 使用流式输出 | 启用 Streaming | 感知延迟降低 70% | 无 |
| 选择更快的模型 | 小模型替代大模型 | 延迟降低 60% | Token 成本降低 |
| 本地缓存 | 缓存常用响应 | 延迟降至 0 | 无额外成本 |

## 如何选择合适的AI模型

实际使用中，需要根据任务复杂度和性能要求选择合适的模型。

### 大参数模型（如 GPT-4、Claude Opus）

**优势**：
- 理解能力强，处理复杂逻辑更准确
- 支持更长的上下文窗口
- 推理能力更强，适合抽象问题

**劣势**：
- 响应速度慢
- 使用成本高
- 本地部署困难

**适用场景**：
- 企业级大型项目
- 复杂业务逻辑
- 架构设计和技术决策

### 小参数模型（如 GPT-3.5、Claude Haiku）

**优势**：
- 响应速度快
- 使用成本低
- 可能支持本地部署

**劣势**：
- 处理复杂任务容易出错
- 上下文窗口较小
- 推理能力相对较弱

**适用场景**：
- 简单重复任务
- 代码格式化
- 快速问答

### 选择建议

- **日常开发**：中等模型（如 Claude Sonnet）平衡性能与成本
- **关键任务**：大模型确保质量
- **批量处理**：小模型降低成本
- **成本敏感**：优先小模型，必要时切换大模型

## MCP（Model Context Protocol）

**概念说明**：
AI 模型与外部数据源连接的标准协议。通过 MCP，AI 可以访问数据库、API、文件系统等资源，突破上下文窗口的限制。

**为什么重要**：
- 让 AI 能够访问实时数据和外部系统
- 解决"AI 不知道你项目最新状态"的问题
- 实现 AI 与企业内部系统的集成
- 需要配置和开发 MCP Server，属于高级应用

**技术架构**：
```
AI 工具（客户端）
    ↓ MCP 协议
MCP Server（中间层）
    ↓ 具体实现
外部数据源（数据库、API、文件等）
```

**实战场景**：

### 场景1：连接数据库
```
需求：AI 直接查询生产数据库分析问题

配置 MCP Server：
{
  "mcpServers": {
    "database": {
      "command": "mcp-server-mysql",
      "args": ["--host", "localhost", "--database", "myapp"]
    }
  }
}

使用效果：
你：查询最近一周注册用户数量的变化趋势
AI：[通过 MCP 执行 SQL] → [分析数据] → [输出趋势报告]

优势：无需手动导出数据，基于实时数据分析
```

### 场景2：连接内部API
```
需求：访问公司内部的 API 文档服务

配置 MCP Server：
{
  "mcpServers": {
    "api-docs": {
      "command": "mcp-server-http",
      "args": ["--base-url", "https://api-docs.company.internal"]
    }
  }
}

使用效果：
你：支付服务的退款 API 参数是什么？
AI：[通过 MCP 查询内部文档] → [提取参数说明] → [输出清晰的文档]

优势：无需离开 AI 工具查阅文档
```

### 场景3：自定义 MCP Server
```python
# 示例：为企业知识库创建 MCP Server
from mcp import Server, Tool

server = Server("knowledge-base")

@server.tool()
async def search_docs(query: str) -> str:
    """搜索企业内部文档"""
    results = knowledge_base.search(query)
    return format_results(results)

@server.tool()
async def get_api_spec(service: str) -> str:
    """获取服务 API 规格"""
    spec = api_registry.get_spec(service)
    return spec

# AI 可以调用这些工具访问企业数据
```

**MCP 的价值**：
- 打破信息孤岛，统一访问各种数据源
- 实时数据分析，基于最新信息决策
- 自动化工作流，减少手动切换工具
- 支持自定义数据源，适配企业内部系统
- 标准化协议，不同 AI 工具可以复用 MCP Server

**技术考量**：
- 需要开发和维护 MCP Server
- 注意数据安全和访问权限控制
- 考虑性能和并发处理能力
- 错误处理和日志记录
- 不是所有 AI 工具都支持 MCP

## RAG（Retrieval-Augmented Generation）

**概念说明**：
检索增强生成。先从知识库检索相关信息，再结合这些信息生成回答，可有效减少幻觉，提供更准确的答案。

**工作流程**：
```
1. 用户提问
"数据库查询慢怎么优化？"

2. 检索阶段
在知识库中搜索相关文档
→ 找到"SQL 性能优化指南"
→ 找到"索引设计最佳实践"

3. 生成阶段
结合检索到的文档生成回答
→ 基于实际文档内容
→ 减少错误和幻觉

4. 返回结果
准确的优化建议 + 文档引用
```

**RAG vs 纯 LLM**：

| 特性 | 纯 LLM | RAG |
|------|--------|-----|
| 知识来源 | 训练数据（固定） | 实时检索（可更新） |
| 准确性 | 可能出现幻觉 | 基于实际文档，更准确 |
| 可追溯性 | 无法引用来源 | 可以显示引用文档 |
| 知识更新 | 需要重新训练 | 更新知识库即可 |

**应用场景**：
- 企业内部知识库问答
- 技术文档查询
- 历史问题解决方案复用
- 项目特定知识管理

## Embedding

**概念说明**：
将文本转换为数字向量的技术。语义相似的文本向量距离更近，是 RAG 系统的核心技术。

**工作原理**：
```
原始文本：
- "如何连接数据库"
- "数据库连接方法"
- "今天天气真好"

转换为向量：
- [0.2, 0.8, 0.1, ...] (数据库连接)
- [0.3, 0.7, 0.2, ...] (数据库连接)
- [0.9, 0.1, 0.8, ...] (天气)

计算相似度：
前两个向量距离近（语义相似）
第三个向量距离远（语义不同）
```

**在 RAG 中的应用**：
```
用户问题："数据库慢怎么办"
↓ Embedding
问题向量：[0.25, 0.75, 0.15, ...]
↓ 相似度搜索
知识库中最相似的文档：
1. "数据库性能优化指南"（相似度 0.92）
2. "SQL 查询调优方法"（相似度 0.87）
3. "索引设计最佳实践"（相似度 0.81）
↓ 提供给 AI
基于这些文档生成回答
```

**常见 Embedding 模型**：
- OpenAI: text-embedding-3-small/large
- HuggingFace: sentence-transformers
- 开源模型: BGE、M3E
