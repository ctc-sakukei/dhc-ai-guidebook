# 540-性能调优与监控

> **难度级别**：⭐⭐⭐ 高级
> **目标读者**：Team Leader，负责性能优化

## 大型代码库优化

### 问题：读取大文件导致响应慢

**优化前**：
```
"请分析整个项目的代码质量"
→ 读取所有文件 → 超时
```

**优化后**：
```
"先列出项目主要模块"
"然后逐个分析auth模块"
"接着分析payment模块"
```

### 问题：上下文过大

**策略1：分层处理**
```
第1步：分析架构（只读核心文件）
第2步：分析模块（按模块拆分）
第3步：汇总结果
```

**策略2：使用Agent**
```
启动多个Agent并行分析：
- Agent1：分析前端代码
- Agent2：分析后端代码
- Agent3：分析测试代码
```

## Token消耗深度优化

### 技巧1：精准读取
```markdown
❌ 浪费Token：
"读取src目录下所有文件"

✅ 精准读取：
"只读取src/auth/Login.tsx"
```

### 技巧2：利用Memory
```markdown
第一次：分析代码 → 消耗1000 Token
记录到Memory：关键设计决策

第二次：查询Memory → 消耗50 Token
```

### 技巧3：增量处理
```markdown
只处理变更部分：
git diff main...feature-branch
```

## 并行任务处理

### 场景：多模块重构

**串行方式**（慢）：
```
1. 重构模块A（5分钟）
2. 重构模块B（5分钟）
3. 重构模块C（5分钟）
总计：15分钟
```

**并行方式**（快）：
```
同时启动3个Agent：
- Agent1：重构模块A
- Agent2：重构模块B
- Agent3：重构模块C
总计：5分钟
```

**实现**：
```
使用Agent工具的run_in_background参数
```

## 缓存策略高级应用

### Prompt缓存

Claude自动缓存：
- CLAUDE.md内容
- Memory内容
- 系统提示

**优化建议**：
- 将稳定内容放在Prompt前面
- 避免频繁修改CLAUDE.md
- 充分利用Memory

### 结果缓存（自建）

```markdown
场景：频繁查询相同信息

方案：
1. 第一次查询 → 保存到Memory
2. 后续查询 → 先查Memory
3. 定期更新Memory
```

## 性能监控指标

### 关键指标

| 指标 | 目标值 | 警戒值 |
|------|--------|--------|
| 平均响应时间 | < 10s | > 30s |
| Token消耗/任务 | < 5k | > 20k |
| 任务成功率 | > 95% | < 80% |
| 上下文使用率 | < 70% | > 90% |

### 监控方法

**手动监控**：
- 记录每次任务Token消耗
- 统计平均响应时间

**自动监控**（推荐）：
```bash
# 分析Claude Code日志
grep "Token" ~/.claude/logs/
```

## 瓶颈诊断方法

### 步骤1：识别慢任务
```
观察哪些任务耗时长：
- 读取文件？
- 生成代码？
- 运行测试？
```

### 步骤2：定位原因
```
可能原因：
- 文件过大
- Prompt过于复杂
- 上下文过大
- 网络问题
```

### 步骤3：针对性优化
```
根据原因选择方案：
- 分批读取
- 简化Prompt
- 清理上下文
- 检查网络
```

## 实战案例

### 案例：大型重构项目

**背景**：重构10万行代码项目

**问题**：单次处理超时

**方案**：
```
1. 按模块拆分（20个模块）
2. 每个模块独立处理
3. 使用Agent并行（5个一批）
4. 总耗时从2小时减少到30分钟
```
