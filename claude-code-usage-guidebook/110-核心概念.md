# Claude Code 核心概念

## 1. CLAUDE.md - 项目级指令文件

CLAUDE.md 是 Claude Code 的核心配置文件，位于项目根目录。它就像是给 AI 助手的"项目说明书"，用于告诉 Claude 关于你的项目的重要信息：

- **项目架构与约定**：技术栈、编码规范、命名约定等
- **工作流程**：测试流程、部署步骤、Git 工作流等
- **特定指令**：你希望 Claude 在这个项目中遵循的特定规则

当 Claude Code 在项目中工作时，会自动读取这个文件，确保它的行为符合项目的特定需求。这大大减少了重复说明项目规则的次数。

## 2. .claude 文件夹 - 持久化配置中心

.claude 文件夹是 Claude Code 的配置和记忆存储目录，通常位于项目根目录或用户主目录下。它包含：

- **memory/**：AI 的长期记忆存储，记录项目经验和用户偏好
- **skills/**：自定义技能脚本，扩展 Claude 的能力
- **hooks/**：事件钩子脚本，在特定操作时自动执行
- **settings.json**：项目级别的配置文件

这个文件夹让 Claude Code 能够"记住"项目的历史经验，并在未来的会话中继续使用这些知识。

### .claude 的详细目录结构

```
.claude/
├── skills/         # 项目自定义技能（推荐）
├── commands/       # 项目自定义命令（⚠️ 已废弃）
├── memory/         # 持久化记忆
└── resources/      # 项目资源文件
```

> **重要提示：** 从Claude Code v2.1.3开始，Command功能已被Skill取代。现有commands虽然能用，但不再维护。

### .claude 文件夹中的核心组件

#### Command（命令）

Claude Code的内置命令，用于控制Claude Code行为：
- `/help` - 显示帮助
- `/init` - 生成CLAUDE.md
- `/clear` - 清空上下文
- `/compact` - 压缩上下文

#### Skill（技能）

项目自定义的可复用工具，通过 `/skill名` 调用。Skill是项目级别的自定义功能，可以根据团队需求创建专属工具。

#### Agent（代理）

Claude Code内置的专门化子任务处理器，拥有独立上下文：
- `Explore` - 代码探索
- `Plan` - 实现计划
- 各种QA工程师agent

#### Memory（记忆系统）

持久化记忆存储，目录结构示例：

```
.claude/projects/[项目名]/memory/
├── MEMORY.md          # 主记忆（自动加载）
├── debugging.md       # 调试经验
├── patterns.md        # 代码模式
└── api-conventions.md # API规约
```

**MEMORY.md特性：**
- ✅ 每次对话自动加载到上下文
- ⚠️ 仅加载前200行，超出部分被截断
- 📌 适合保存架构要点、常见问题、代码规范

### Memory系统实战案例

#### 案例1：记录API设计规范

**场景**：团队在开发过程中总结出REST API的统一规范，希望Claude Code在生成API代码时自动遵循。

**解决方案**：在`.claude/memory/api-conventions.md`中记录：

```markdown
# API设计规范

## 响应格式
所有API响应统一使用以下格式：
```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

## 错误码规范
- 0: 成功
- 1000-1999: 参数错误
- 2000-2999: 业务逻辑错误
- 5000-5999: 系统错误

## 命名规范
- 端点使用复数形式：/api/users, /api/orders
- 使用kebab-case：/api/user-profiles
- 版本号前缀：/api/v1/users
```

**效果**：之后让Claude Code生成任何API相关代码时，都会自动遵循这些规范，无需每次重复说明。

#### 案例2：记录调试经验

**场景**：项目中遇到过一个特定的性能问题，经过排查找到了解决方案，希望Claude Code在后续遇到类似问题时能想起这个经验。

**解决方案**：在`.claude/memory/debugging.md`中记录：

```markdown
# 调试经验记录

## 数据库查询性能问题（2024-03-08）

**问题**：用户列表页面加载缓慢，查询耗时超过3秒

**排查过程**：
1. 检查SQL日志，发现N+1查询问题
2. 用户表关联了多个表但未使用JOIN

**解决方案**：
```typescript
// 错误做法
const users = await User.findAll();
for (const user of users) {
  user.profile = await Profile.findOne({ userId: user.id });
}

// 正确做法
const users = await User.findAll({
  include: [{ model: Profile }]
});
```

**经验教训**：
- 涉及关联查询时优先使用JOIN
- 开发环境开启SQL日志便于排查
- 列表页面注意分页和索引优化
```

**效果**：当再次需要处理类似的查询优化时，Claude Code会参考这个经验，避免重复踩坑。

#### 案例3：记录项目架构演进

**场景**：项目从单体架构迁移到微服务架构，新旧代码并存期间需要让Claude Code了解架构变化。

**解决方案**：在`.claude/memory/MEMORY.md`中记录：

```markdown
# 项目架构演进记录

## 当前架构状态（2024-03）

项目正处于架构迁移期：
- ✅ 已迁移：用户服务、订单服务
- 🔄 迁移中：支付服务
- ⏳ 待迁移：商品服务、库存服务

## 代码组织规则

### 新微服务代码（推荐）
位置：`/services/[服务名]/`
- 独立数据库
- gRPC通信
- 使用Kubernetes部署

### 旧单体代码（逐步废弃）
位置：`/legacy/`
- 共享数据库
- REST API
- 不要在此添加新功能

## 开发注意事项
- 新功能一律在微服务中实现
- 修改旧代码时评估是否适合迁移
- 跨服务调用使用服务网格
```

**效果**：Claude Code在修改代码时会：
- 知道哪些模块是新架构，哪些是旧架构
- 在合适的位置添加新功能
- 建议何时应该迁移旧代码

### Agent、Skill、Command的区别

| 特性 | Agent | Skill | Command |
|-----|-------|-------|---------|
| 定义 | 内置子任务处理器 | 项目级工具 | 系统控制命令 |
| 上下文 | 独立上下文空间 | 共享主上下文 | 无上下文 |
| 适用场景 | 复杂多步骤任务 | 标准化开发任务 | 控制Claude Code行为 |
| 调用方式 | 对话中指定或自动 | `/skill名` | `/命令名` |
| 示例 | Explore, Plan | test:run | /clear, /help |

## 3. 上下文(Context) - Claude 的工作视野

上下文是 Claude Code 在执行任务时能够"看到"和"理解"的所有信息。它就像是 AI 的工作台，决定了它能利用哪些信息来完成任务：

- **对话历史**：当前会话中的所有交互内容
- **文件内容**：通过 Read、Grep 等工具读取的文件
- **项目配置**：CLAUDE.md、memory 等配置信息
- **系统信息**：Git 状态、环境变量、当前目录等

理解上下文管理对于高效使用 Claude Code 至关重要，因为它直接影响 AI 的决策质量和响应速度。合理的上下文管理可以让 Claude 更准确地理解需求，做出更好的决策。

### 上下文工作原理

Claude Code本质上是**无状态**的，不会真正"记住"对话。所谓"记忆"是每次都附加历史对话。

**对话示例：**

```
第一轮：
你：你好，我叫Jason
API发送：你好，我叫Jason
Claude Code：你好，Jason

第二轮：
你：我的名字是什么？
API发送：
  <历史>
    用户：你好，我叫Jason
    Claude Code：你好，Jason
  </历史>
  <新指令>我的名字是什么？</新指令>
Claude Code：<思考>从历史中看到用户叫Jason</思考>
    你的名字叫Jason
```

**关键影响：**

- 对话越长 → 上下文越大 → Token消耗越多 → 响应越慢
- 上下文达到100% → 自动执行`/compact`压缩
- 压缩会**永久丢失**部分信息

## 4. 三者之间的关系

这三个概念是相互配合、层层递进的关系：

```
CLAUDE.md (项目规则)
    ↓
.claude 文件夹 (持久化存储)
    ↓
上下文 (工作时的信息集合)
```

- **CLAUDE.md** 定义了"规则"：告诉 Claude 这个项目应该怎么工作
- **.claude 文件夹** 提供了"记忆"：存储了 Claude 从项目中学到的经验
- **上下文** 是"当前视野"：Claude 当前能看到的所有信息（包含上面两者）

简单来说：**CLAUDE.md 是项目的长期规则，.claude 是积累的经验，上下文是当前工作时能用到的信息**。

## 5. 一个简单的例子

假设你正在开发一个 Node.js 项目，想让 Claude Code 帮你添加一个新功能：

### 场景设置

你的项目中有：
- **CLAUDE.md** 写着：
  ```markdown
  # 项目规则
  - 使用 TypeScript
  - 测试框架是 Jest
  - 每个新功能都要写单元测试
  ```

- **.claude/memory/** 中记录着：
  ```markdown
  用户喜欢使用箭头函数而不是 function 关键字
  项目中的 API 调用统一使用 axios
  ```

### 工作流程

1. **你说**："帮我添加一个获取用户信息的 API"

2. **Claude 的上下文包含**：
   - 你刚才说的话（对话历史）
   - CLAUDE.md 中的项目规则
   - memory 中的用户偏好
   - 当前的 Git 状态、项目结构等

3. **Claude 会**：
   - 使用 TypeScript（遵循 CLAUDE.md）
   - 使用 axios 发送请求（遵循 memory）
   - 用箭头函数编写代码（遵循 memory）
   - 自动创建对应的 Jest 测试（遵循 CLAUDE.md）

4. **结果**：Claude 生成的代码完全符合你的项目规范和个人习惯，无需反复说明。

### 关键点

这个例子展示了：
- **CLAUDE.md** 确保了技术栈的一致性
- **.claude/memory** 记住了你的偏好
- **上下文** 让 Claude 能综合考虑所有信息做出决策
